# FTHR-009: Persona-driven local orchestrator and consult_brain nested loop

## Description
This is the core topology reorientation. Today `Loop.run_turn` routes the *whole* turn to whichever tier `Router.select(tools_available=...)` picks — when tools exist, that's the remote tier, so the remote model addresses the user directly (the identity-leak root cause) and directly holds `wikipedia_search`. This feather inverts that: the top-level turn is always served by the **local** tier, carrying a Calcifer persona system prompt, and its only tool is `consult_brain(query)` — calling it runs a **nested** ReAct loop on the **remote** tier over the real data tools (`wikipedia_search`), whose findings return as an observation the local orchestrator incorporates into its persona-voiced answer. `wikipedia_search` moves off the top-level loop entirely; it is only reachable from inside a consult. A `BrainError` (FTHR-008) raised during a consult becomes a graceful observation, not a crashed turn. Verified via I/O by real code reading (`hearth/loop.py`, `hearth/brain/router.py`, `hearth/app.py`, `hearth/config.py`, `hearth/tools/registry.py`, `hearth/brain/base.py` on the current tree — nest docs predate this restart and are not authoritative for these files).

## Approach
- **`run_react_rounds` seam.** Signature: `async def run_react_rounds(*, brain, messages, tools, dispatch, round_cap, log, session_id, turn_id, emit, label_for) -> BrainResult`. It performs the first `brain.complete(messages, tools=tools)`, then the existing per-round mechanic unchanged (append the assistant tool-call message, `ToolActivity` start/end via `label_for(call.name)`, log `tool_call`/`observation`, the tool-failure-to-observation guard try/except around `dispatch(...)`, append the tool message, re-`complete()`) capped at `round_cap`, and the existing "ran out of rounds" text fallback. This one function is called by `Loop.run_turn` (brain=local, tools=`[consult.SPEC]`, dispatch=`self._consult_dispatch`) and by `BrainConsult.__call__` (brain=remote, tools=wikipedia specs, dispatch=`tool_registry.dispatch`) — no duplicated ReAct logic.
- **Two registries, two roles**, per the plan: orchestrator registry (consult_brain only, built fresh — not `ToolRegistry`, since that class's `dispatch` is frozen to wikipedia per FTHR-006) vs. the existing `ToolRegistry` reused unchanged as the brain-side wikipedia registry inside `BrainConsult`.
- **`emit` threads through the nested loop too** — `run_react_rounds` takes the same `emit` for both call sites, so the client sees content-free `ToolActivity(start/end, "consult")` for the orchestrator's own call and `ToolActivity(start/end, "search")` for the nested wikipedia call, in sequence. `serialize()` (`hearth/veneer/protocol.py`) is untouched — still only phase+label cross the wire.
- **`Loop._consult_dispatch`** is the glue: `run_react_rounds`'s `dispatch(name, args)` contract is `(name: str, args: dict) -> str`, matching `ToolRegistry.dispatch`; `Loop._consult_dispatch` closes over the current turn's `session_id`/`turn_id`/`emit` and forwards `args["query"]` to `self._consult(session_id, turn_id, args["query"], emit)`.
- **Round-cap/timeout table** (distinct caps, per the approved plan):

  | Scope | Round cap | Timeout |
  |---|---|---|
  | Orchestrator consult rounds (local emits `consult_brain`) | `agent.max_consult_rounds` (new, 3) | `agent.turn_timeout_s` (existing, 45s) wraps the whole orchestrator loop |
  | Nested brain tool rounds (remote emits `wikipedia_search`) | `agent.max_tool_rounds` (existing, 3, repurposed) | `agent.consult_timeout_s` (new, 30.0) wraps one consult |

  `consult_timeout_s < turn_timeout_s` so one slow consult can't eat the whole turn, while the outer `wait_for` still hard-bounds a chain of consults.
- Preserve PLM-001's local-only fallback structurally: `Loop` calls `router.brain_available()` once per turn (not per-call) to decide whether the orchestrator registry offers `consult_brain` at all; when false, `tools=None` and the turn is pure local chat — no crash, no dangling tool offer.
